State flags

Internal flags that decide which branch of the reply formatter fires. Useful when reasoning about why a particular reply wording was chosen.

FlagSet when…Effect
isShunterCallCaller name matches /shunter/i, or driver pattern from sidingsAdapts shunter-style permission and call-back wording
isSignallerCallSig advisory or early-running signaller call detectedSwitches to peer-to-peer phrasing
isThirdPartyCallCaller is not Driver/Signaller/Shunter and no headcodeAll replies become 3rd-person about the driver
isUnlitSignalDriver message contains "unlit signal"Enables unlit-state pass authorisation wording
isAdverseAspectMessage contains "adverse change of aspect""Continue after speaking" → "obey all other signals" override
isRouteQueryDriver message matches route query parserUses route-control wording for replies
isReadyToDepartMessage matches the ready-to-depart parserAcknowledge with "Signaller out" close
isRedSignalCallMessage matches "waiting at red signal" or "at signal X displaying red"Every reply opens with "Hello [head] standing at signal [sig]."
examiningTrains[head]Signaller previously sent any examine-related reply for this trainSwitches "I also require you to examine" → "Please continue to examine"
All flags reset between calls. examiningTrains and trainSignalCache persist across calls within the same SimSig session and clear on disconnect.
